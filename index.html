The Core Problem

Operational Systems Were Designed
to Process Transactions. Not to Answer Questions.

ERP, CRM, billing, field operations, HR — each system was engineered to handle a specific workflow as fast and reliably as possible. The data structure inside reflects that purpose.

✗ Column names reflect internal system logic — not business meaning
✗ The same column name may mean four different things across tables
✗ Installed modules default to "OK" — whether or not they were ever configured
✗ Tables designed as contact registries get queried as customer databases
✗ Data about the same entity is split across systems with incompatible IDs
✗ 64% of schema tables can be empty — creating a minefield for any direct query

The default approach

An analyst connects a BI tool to the operational database, finds the tables that seem relevant, writes queries, and builds dashboards. The results look authoritative — they have specific numbers, dates, and trends.

The hidden problem

The database never errors. It always returns a number. If that number is wrong because a column was never activated, or a table is a registry not a customer list, or IDs don't match across systems — the dashboard does not know. And neither does the manager reading it.

In the sample database: 1,027 tables, 64% with zero rows. The system had modules for OLT hierarchy, slot management, GPON ports, SLA tracking — all installed, all empty, all returning zero without warning.

The Hardest Trap to Catch

A Metric That Shows a Real Trend
Because the Data Gets Filled In Over Time

Some wrong metrics are obvious — a column that's always zero. Others are far more dangerous: metrics that show movement, trend lines, apparent growth — but the movement is caused by changing data-entry habits, not by the business changing.

What actually happened

The field used to measure this was sparsely filled. Teams started filling it more consistently in April. The subscriber base did not change.
Why this is the most dangerous trap: A wrong static number (like 27,115 vs 19,194) can be challenged in a meeting. A trend line that shows "growth" gets used in forecasts, targets, and investment decisions — and no one questions a positive trend.

Why Approach A Cannot Detect This

✗ The field exists in the database — it is a valid column with real values
✗ The trend is mathematically correct — April does have more filled values than February
✗ There is no error, no null count warning, no anomaly flag
✗ The only way to detect it is to cross-reference the fill rate against a second source — which requires understanding the data architecture, not just the data

What Systematic Investigation Finds

✓ The field (partner_id on work orders) is only 22.3% populated overall
✓ Subscriber identity in this workflow is actually tracked via contract_id (66.8%) and a port string (53.7%) — neither used by the dashboard
✓ The fill rate variation is documented as a caveat — the metric is retired and replaced with the correct identifier

The Pipeline in Practice

Each Layer Solves a Specific Class of Problem

🧹

Staging — "What does this column actually mean?"

The staging layer is where column names are corrected, misleading labels are renamed, dead fields are excluded, and population filters are applied. sap_id renamed to reflect its actual source (CRM vs network system). res_partner filtered to subscribers only. Every model is documented in YAML.

⚙️

Intermediate — "What does this mean for the business?"

Business logic is applied once and shared by everything downstream. "Completed job" is defined here — not in each dashboard independently. Cross-system identifiers are resolved. The notes field is parsed. Computed flags like is_placeholder_zone are derived.

✅

Mart — "Is this number trustworthy enough to present?"

The mart is where grain is validated (row count = unique IDs), automated tests run, and caveats are documented. If network coverage is 88.5% for one job type and 45.5% for another, that is stated explicitly on the column — not hidden behind a confident-looking average.

The Principle: Business Logic Lives Once

In the old approach, each dashboard re-implements the same rules from scratch. The definition of "completed" is written four different ways across four different reports.

In the pipeline approach, "completed" is defined in one place in the intermediate layer. Every dashboard inherits it. Change the definition once — every report reflects the change on the next run.

Why automated tests matter

✓ Grain test: 10,554 rows = 10,554 unique work order IDs. Zero duplicates. Verified arithmetically — not assumed.
✓ Relationship tests: every foreign key verified against its parent table — no silent join failures
✓ Vocabulary tests: stage types, task types — if a new value appears that the pipeline doesn't handle, it surfaces as a test warning before the dashboard
✓ 32 tests, 0 failures: the pipeline delivered a clean build before any number was presented to management

The Broader Point

This Is Not a Fix for One Database.
It Is a Practice for Every Source.

The Lightspeed database is one example. The traps found there — a contact registry mistaken for a customer table, inactive modules reporting defaults, data across systems with incompatible identifiers — are not unique to this company or this software.

They are structural properties of how operational systems are built. They appear in Salesforce, SAP, Oracle, custom-built systems, and every legacy database. The names change. The patterns do not.

The repeatable pattern — 5 steps per source

1

Probe

Systematically investigate the source before writing a single report query. Document what each column actually contains, not what it's named.

2

Stage

Clean, rename, and document every field. Apply population filters. Remove dead columns. One model per source table.

3

Transform

Apply business logic once. Resolve cross-system identifiers. Derive computed metrics with explicit definitions.

4

Validate

Test grain, relationships, vocabularies. Document caveats. Zero errors before any number is presented.

5

Serve

All BI tools read from the validated mart only. No raw-table queries. Definitions are in the pipeline, not in the dashboard.

What this gives the organization — permanently

✓ One number per KPI — the same answer in every report, every meeting, every team
✓ Failures as alerts, not surprises — automated tests surface data problems before dashboards, not during board presentations
✓ Cross-system questions answered — the pattern connects sources; each added source extends what is possible
✓ Onboarding in hours, not months — documented pipeline vs. undocumented raw table queries
✓ Audit trail — every definition change is tracked; every number is traceable to its source
✓ Scalable — incremental processing handles growing data volumes without rebuilding from scratch

What this approach does NOT claim

⚑ It does not fix data that was never collected — gaps in source systems remain gaps
⚑ It does not eliminate the need for domain expertise — business definitions still require business input
⚑ It does not make every metric 100% complete — but it documents exactly what the coverage is
Recommendation

Start With One Source. Build the Pattern.
Extend to Every System.

The proof of concept is complete on the field operations database. The pipeline works. The pattern is documented. The next step is validation and extension — not rebuilding from scratch for each new source.

1

Validate with the business owners

Review the validated field operations KPIs with the operations and CC teams. Confirm metric definitions match operational reality. Two weeks.

2

Retire raw-table reporting for this domain

Once confirmed, all field operations reporting reads from the validated pipeline. Raw-table Grafana queries are archived. The first source is declared the reference implementation.

3

Apply the pattern to the next priority source

CRM, billing, or HR — whichever unlocks the highest-value cross-system question. Each new source follows the same 5-step probe-stage-transform-validate-serve cycle.

4

Connect sources as the foundation grows

As each source is added, cross-system metrics become available. Subscriber lifecycle. Cost-per-zone. Technician attendance vs dispatch. Questions that are currently unanswerable become routine.

The compounding return

The first source is the hardest — it establishes the foundation, the conventions, and the tooling. Each source added after that is faster. The cross-system connections that required months of investigation become a configuration, not a project.
