The Situation

The Company Has Multiple Operational Databases

Each system of record serves a different operational need. Dashboards and reports are built by querying these raw operational tables directly — different analysts, different queries, for the same business questions.

The result is a fragmented landscape where the same metric produces different numbers depending on who built the report and which table they queried.

Odoo ERP — odoo_earth

Field operations, work orders, technician dispatch — 1,027 tables, 64% empty

Creatio CRM

All subscriber accounts, CC tickets, installation orders — external system

FMS (Fiber Management System)

Contracts, network topology, billing, subscriber IDs — external system

HRIS

Employee records, org chart, payroll — external system

Odoo is not a standalone ERP. It is an operational middleware — dispatching field teams, logging work orders, and syncing to FMS and Creatio. Querying it without understanding its role produces structurally incomplete answers.

Placeholder Zones — Ghost Geography

12 zones with codes ending 8888 or 8822 are internal placeholders for uncommissioned network areas:

1,775

work orders (16.8% of total) in placeholder zones — including 1,496 completed installations. Zone-level KPIs mix real and ghost geography without this flag, making regional performance analysis incorrect.

🔍

Description Field — Structured Data Treated as Free Text

The maintenance_request.description field was excluded as "unstructured free text." A probe revealed it contains a structured CC dispatch template with Zone, FAT node, and GPS coordinates. Excluding it loses network attribution for 75.3% of all work orders. After parsing: 96.7% FAT/Zone coverage.

👤

HRIS Employee Codes — 78.5% Are a Python Import Bug

hr_employee.emp_id should link employees to the HR system. Probe: 2,523 of 3,215 employees have values like False1, False2 — Python False imported as string. Only 692 (21.5%) have genuine HRIS IDs. Cross-system HR analytics cover fewer than 1 in 5 staff.

The Core Principle

Fix Business Logic Once. It Applies Everywhere.

The single most impactful shift: business logic moves from inside dashboard queries to inside transformation models. Change a rule once — rebuild — every downstream consumer reflects the change automatically.

Centralized Definitions — Before vs After

Before: "Completed job" = stage_id = 9 in Dashboard A, stage_type = 'done' in B, stage name join in C, missing in D. Four queries, four possible answers.
After: is_done = TRUE defined in stg_odoo__maintenance_stage.sql. All 42 panels use this column. Change the definition → one file → one rebuild.
Before: Cash collection = collection = TRUE (always FALSE — 0 results company-wide).
After: is_cash_collected = (collected_amount_iqd > 0) — defined once, correctly identifies 2,551 jobs.

The Rollout Pattern — 5 Steps Per Source

1. Probe — systematic SQL probes against the raw DB; classify tables; document fill rates
2. Stage — clean models per table; cast types; rename misleading columns; add YAML docs
3. Integrate — intermediate models apply business logic; resolve FKs; derive flags
4. Mart — one fact/dimension per domain; grain validated; tests pass before publish
5. Dashboard — BI tool reads mart only; KPI definitions live in dbt, not in Grafana

Additional Value

What dbt Unlocks When Applied to Every Source

🔔

Automated Data Quality Monitoring

171 tests run on every build. If FMS stops syncing, a test fails — before a wrong number reaches a dashboard. Quality issues become alerts, not surprises.

🔐

PII Management

Subscriber names, phone numbers, and port strings are flagged in the mart YAML. Access control applies at the mart layer — analysts see aggregates, not raw contact data.

🔗

Cross-Source Conformed Dimensions

FMS uses a numeric subscriber ID; Creatio uses a UUID; Odoo uses res_partner.id. A dbt dimension table maps all three — enabling true end-to-end subscriber analytics.

🌐

Environment Separation

Dev, staging, and prod schemas are configuration — not code. Analysts test new metrics in dbt_dev_* without touching the board's live dashboards.

⚡

Incremental Refresh at Scale

Action logs: 39,343 rows. Attendance raw: 141,601 rows. As sources grow to millions, incremental models process only new data — keeping refresh times fast.

🚀

Onboarding in Hours, Not Weeks

📋

Full Audit Trail

Every metric definition change is a git commit. When a KPI shifts between quarters, the git log shows exactly what changed, when, and who approved it.

🤖

AI and Semantic Layer Ready

Consistent naming and column documentation makes marts directly consumable by LLM analytics tools. Natural-language queries resolve to the same trusted columns every dashboard uses.
